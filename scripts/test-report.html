1. Overview

The backend is covered by a Jest unit-test suite. Every test is a pure unit test: it runs in-process against mocked dependencies, with no database, no Redis, and no running server. This makes the suite fast (~16 seconds), deterministic, and safe to run in any environment — local machines and the CI runner alike — without provisioning infrastructure.

The suite delivers three kinds of coverage: API service-behavior testing — validating the business logic that powers the API endpoints (voting, reporting, aspirant registration, meetings, visits, ratings, issue hand-raises); security & data-privacy testing — ensuring an aspirant's private contact details are never returned by the API unless the matching permission is enabled; and module wiring checks across all 22 modules.

Tests live next to the code they cover (*.spec.ts beside each source file) and are organised into two complementary layers.

4. API service-behavior coverage (7 services, 62 tests)

AspirantsService — 34 tests

Contact privacy, profile lifecycle, meetings, visits, and ratings.

  • Contact privacy: hides phone from anonymous viewers and other users when allowPhone=false; shows it to the owner regardless; hides whatsappNumber from non-owners when allowWhatsapp=false; shows both to everyone when both flags are true.
  • register(): rejects unauthenticated callers; rejects a phone already used by another user; rejects a whatsapp number already used by another aspirant; rejects a duplicate active profile; creates when all checks pass.
  • updatePermissions(): rejects when the aspirant is not owned by the caller; updates only the provided flags and returns them.
  • withdrawAspirant(): rejects when the user has no profile; blocks withdrawal while voting is open for the same election; otherwise withdraws and reverts the role to voter.
  • Meetings — createBooking(): 404 when the aspirant is missing; otherwise saves a pending booking carrying the voter's message and preferred time. respondToMeeting(): 404 when the meeting is missing; records the RSVP and returns refreshed attending / not-attending counts.
  • Visits — createVisit(): 404 when the aspirant is missing; persists the visit (the constituency notification is best-effort and never blocks the save). respondToVisit(): creates a new RSVP or flips the existing one in place without duplicating.
  • Ratings — rateMeeting() / rateVisit(): 404 when the activity is missing; create a 1–5 rating tied to the activity's aspirant, and are re-ratable (a repeat updates the value in place). rateContact(): eligibility-gated (the voter must first have pressed the aspirant's phone / WhatsApp button) and one-time (a second attempt is rejected).

VotesService — 6 tests

castVote() — the core voting rules.

5. Security & data-privacy testing

A dedicated group of tests treats an aspirant's contact information as protected personal data and verifies the API exposes it only with explicit consent. An aspirant controls visibility through the allowPhone and allowWhatsapp permission flags; when a flag is not enabled, the corresponding value must never be returned. These tests guard against accidentally leaking phone / WhatsApp numbers to voters or other users, and confirm access control across distinct viewer identities.

Scenario (permission flag = off)Expected outcome — enforced by test
Anonymous viewer requests an aspirantphone is omitted entirely — the value never leaves the server (not blanked)
A different logged-in user requests the aspirantphone is omitted
The owner requests their own profilephone is included — the owner's profile screen relies on it
allowWhatsapp off, non-owner viewerwhatsappNumber is omitted; the owner still sees it
Both permission flags ONBoth phone and whatsappNumber are returned to everyone

The permission flags themselves remain in the response so the client knows which contact actions to offer — only the underlying values are withheld. This is verified as part of the 34 AspirantsService tests in section 4.

6. CI/CD integration

The suite runs automatically in GitHub Actions on every pull request and every push to main and staging (.github/workflows/ci.yml). The validate job runs four steps in order:

lint  →  typecheck  →  npm test --runInBand  →  build

Because the tests require no external services, they run on a plain ubuntu-latest runner with only npm ci — no Postgres, Redis, or environment configuration. Any failing test fails the check and blocks the merge (where branch protection requires CI). The deploy workflow (deploy.yml) builds and ships to EC2 only after validation.

7. Running the tests locally

# install once, then run the full suite
npm install
npm test

# watch mode (re-run on change) and coverage
npm run test:watch
npm run test:cov

# target a file, folder, or test name
npx jest src/aspirants
npx jest aspirants.service.spec.ts
npx jest -t "rateContact"
No setup required. The tests do not read .env, connect to a database, or start a server. A fresh clone with npm install can run the entire suite immediately.
